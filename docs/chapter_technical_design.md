# Глава 3. Техническое проектирование и реализация модуля ИИ

## 3.1. Выбор средств разработки

Для реализации системы AutoAssess выбран следующий технологический стек:

| Компонент | Технология | Обоснование |
|-----------|------------|-------------|
| Информационная система | Python 3.11, FastAPI, SQLAlchemy, Alembic | Быстрая разработка REST API, миграции схемы БД |
| СУБД | PostgreSQL 15 (продакшн), SQLite (разработка/E2E) | Соответствие требованиям диссертации |
| Клиентская часть | React 18, TypeScript, Vite | SPA, типизация, модульная UI |
| Модуль ИИ (инференс) | Mistral 7B Instruct v0.1 (GGUF Q4_K_S), llama-cpp-python | Локальное развертывание без облачных API |
| Обучение адаптеров (CPU) | scikit-learn, pandas, joblib | Дообучение голов адаптеров на имеющихся датасетах без GPU |
| Контейнеризация | Docker Compose | Изоляция сервисов postgres, backend, ai-service, frontend |

Базовая модель Mistral 7B остаётся замороженной; обучаются только специализированные «головы» адаптеров (классификаторы и тематические профили шаблонов).

## 3.2. Архитектура программного комплекса

Система построена по микросервисной схеме (рис. 3.1 — см. `docker-compose.yml`):

1. **Frontend** — интерфейс преподавателя, администратора и студента.
2. **Backend** — бизнес-логика, JWT-аутентификация, оркестратор проверки.
3. **AI-service** — четыре адаптера оценки, загрузка GGUF, применение обученных моделей.
4. **PostgreSQL** — хранение пользователей, шаблонов, сессий, ответов, результатов ИИ.

Поток проверки: студент отправляет бланк → backend вызывает `POST /evaluate` модуля ИИ → результаты сохраняются в `ai_check_result.details` (JSON по каждому адаптеру).

## 3.3. Проектирование обучения адаптеров

### 3.3.1. Адаптер релевантности (Pfeiffer, слои 9–12)

**Отдельный обучающий датасет не используется.** Адаптация выполняется при загрузке лекции (PDF/DOCX) к шаблону:

1. Извлечение текста лекции (`backend/app/services/lecture_text.py`).
2. Формирование тематического вектора **T** — частотный анализ терминов (`lecture_topics.py`).
3. Сохранение профиля: `ai-service/models/adapters/relevance/template_{id}.json`.
4. При проверке: **R = f(Q, S, L)** с учётом T и текста лекции (`relevance_engine.py`).

Это соответствует §2.2.4.2 диссертации: обучение на лекционном материале, прикреплённом к шаблону.

### 3.3.2. Адаптер нормативности (Parallel, слои 7–12)

**Датасеты** (папка `dataset/`):

- `russian_gec_dataset_final (1).csv` — пары «ошибка → исправление»;
- `LORuGEC.xlsx` — GEC с привязкой к грамматическим правилам;
- `in_domain_train.csv` — бинарная грамматическая приемлемость предложений.

**Подготовка:** `training/prepare_datasets.py` → `training/processed/normativity.jsonl` (59 785 записей).

**Обучение:** TF-IDF (1–2-граммы, до 50 000 признаков) + логистическая регрессия с `class_weight=balanced`. Модель: `ai-service/models/adapters/normativity.joblib`.

**Метрики на тестовой выборке (15 %):**

| Метрика | Значение |
|---------|----------|
| Accuracy | 0,802 |
| F1-score | 0,822 |
| Обучающая выборка | 50 817 |
| Тестовая выборка | 8 968 |

**Инференс:** текст ответа разбивается на предложения; для каждого вычисляется вероятность грамматической корректности; итоговый балл — среднее по предложениям; слабые предложения попадают в `errors[]`.

### 3.3.3. Адаптер логики (Houlsby, слои 1–12)

**Датасет:** `LiDiRus.jsonl` — 1 104 пары предложений, метки `entailment` / `not_entailment`, типы логических конструкций.

**Подготовка:** `training/processed/logic.jsonl`.

**Обучение:** TF-IDF + логистическая регрессия на объединённом тексте пары предложений. Модель: `logic.joblib`.

**Метрики на тестовой выборке:**

| Метрика | Значение |
|---------|----------|
| Accuracy | 0,542 |
| F1-score | 0,608 |
| Обучающая выборка | 938 |
| Тестовая выборка | 166 |

Относительно низкая accuracy объясняется переносом задачи NLI (два предложения) на проверку соседних фрагментов одного ответа студента; при появлении размеченных ответов с `logic_errors` метрики можно улучшить дообучением.

### 3.3.4. Адаптер корректности (Houlsby)

На текущем этапе используется режим **LLM-as-a-Judge** (zero-shot / с GGUF): сравнение **S** с эталоном **E**. Отдельный датасет корректности в `dataset/` отсутствует; после накопления экспертных оценок в AutoAssess планируется дообучение LoRA на GPU.

## 3.4. Реализация модуля ИИ

Структура `ai-service/`:

```
ai-service/
├── app/
│   ├── adapters/          # relevance, correctness, normativity, logic
│   ├── llm/               # loader, inference, *_engine.py
│   ├── ml/                # trained_models.py — загрузка joblib
│   └── routers/           # /evaluate, /health
└── models/
    ├── mistral-7b-....gguf
    └── adapters/
        ├── normativity.joblib
        ├── logic.joblib
        └── relevance/template_{id}.json
```

Приоритет инференса:

1. Если загружена обученная головка (normativity/logic) — используется она.
2. Иначе, если загружен GGUF Mistral — промпт LLM-as-a-Judge.
3. Иначе — эвристический fallback.

Эндпоинт `/health` возвращает флаги `trained_heads` и `model_loaded`.

## 3.5. Интеграция с информационной системой

- Загрузка лекции: `POST /api/templates/{id}/lecture` (PDF, DOCX) → извлечение тем → профиль релевантности.
- Проверка: `Orchestrator` передаёт `template_id`, текст лекции в `context_materials`, веса из шаблона.
- Результаты: JSON `details` с полями по каждому адаптеру (scores, errors, topic_mismatches и т.д.).

## 3.6. Запуск обучения и развёртывания

```powershell
# Обучение адаптеров на CPU
.\training\run_training.ps1

# Локальный E2E
.\scripts\setup_and_run_e2e.ps1

# Docker (при установленном Docker Desktop)
docker-compose up --build
```

## 3.7. Выводы по техническому проектированию

1. Реализован полный контур: датасеты → подготовка → обучение → интеграция в ai-service.
2. Адаптер релевантности адаптируется **per template** при загрузке лекции без отдельного корпуса разметки.
3. Адаптеры нормативности и логики дообучены на открытых русскоязычных корпусах с приемлемым качеством (F1 нормативности 0,82).
4. Корректность остаётся на Mistral zero-shot до появления экспертного датасета.
5. Архитектура допускает замену sklearn-голов на полноценные LoRA-веса Pfeiffer/Houlsby/Parallel при наличии GPU.
